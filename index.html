At first glance, lab science and software testing seem totally different—one's all chemicals and lab equipment, the other’s all code and computers. But the core problem-solving approach is actually pretty similar: figure out what went wrong, identify likely causes, investigate, try a solution, and evaluate the results. Debugging code even has some perks—unlike experiments, it usually has error logs and call stacks. It's also faster and less resource-heavy to test a solution in code than to rerun a hands-on experiment.

Conducting a lab experiment is also a lot like writing and executing a test case. Both involve documenting and following a precise series of steps while understanding each step’s purpose and expected outcome. This helps you spot any unexpected results and determine if it's due to human error, a glitch, or an issue with the procedure. In software QA, it also includes figuring out if the issue is with the software itself or the test. Knowing the system well also helps you decide when and how to adjust things mid-process to stay on track.

When I started coaching at Momentum Gymnastics, I noticed a major gap: there wasn’t a comprehensive skills list for each class, which made training new coaches more challenging. To address this, I decided to build a web application that catalogs skills, links them to specific classes, and provides drills for teaching and improving each skill.

It included login functionality and database interactions for searching, adding, and updating entries, so I customized it for our needs and deployed it to collect feedback from other coaches. I’m now working on a formal test plan and implementing automated testing to enhance the app’s maintainability.
